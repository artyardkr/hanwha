## PART 1. 디지털 전환과 RWA의 필연성

### 핵심 저자

### 🔑 핵심 개념

**RWA (Real World Asset)란?**
- **정의**: 부동산, 채권, 주식, 예술품, 법정화폐 예치금 등 전통 금융자산을 블록체인 상의 디지털 토큰으로 표시
- **기능**: 토큰에 해당 자산에 대한 청구권/소유권 부여
- **효과**: 자산의 거래·정산·권리 이전 자동화

**주요 기관의 입장**

**BIS (국제결제은행, 2025년 연차보고서)**
> "토큰화는 기존 금융을 개선함과 동시에 새로운 것을 가능케 하는 다음 단계 진화(Tokenisation represents the next logical progression in the evolution of the monetary and financial system)"

- 차세대 통합 금융 인프라 구상
- 통합 원장(Unified Ledger): 토큰화된 중앙은행 준비금, 상업은행 예금, 국채 등을 하나의 원장에 통합

**SEC (미국 증권거래위원회) 위원장 폴 앳킨스**
> "토큰화가 시장 전반의 효율성을 높이는 다음 단계(Tokenization is the next step in order to have much more efficiency)"

### 1.1 RWA와 온체인 금융: 온체인에서 구현되는 Trad-Fi

**시장 규모** (2025년 10월 기준, rwa.xyz)
- 퍼블릭 블록체인 상 RWA (스테이블코인 제외): **337억 달러**
  - **사모신용(Private Credit)**: 17.6억 달러 (최대 비중)
  - **미국 국채**: 8.4억 달러
  - **금·부동산** 등 기타

**온체인 금융의 구조적 특징**
- 거래의 기록·결제·정산·보관이 하나의 네트워크에서 처리
- 실물자산이 토큰 형태로 블록체인에 기록
- 전통 금융의 주요 기능들이 온체인 환경에서 재구현

#### 1.1.2 예금, 대출 (Deposits & Loans)

**온체인 대출의 작동 원리**
```
1. 유동성 공급자(Lender): 디지털 자산을 유동성 풀에 예치 → 이자 수익
2. 차입자(Borrower): 자산을 담보로 대출
3. 스마트 컨트랙트: 자동으로 대출/상환/담보 관리/이자율 조정
```

**전통 금융과의 차이**
- 중앙 중개기관(은행) 불필요
- 신용평가 기관, 복잡한 서류 심사 불필요
- 담보 자산 가치 기반 즉각적 대출
- 전 세계 누구에게나 개방

  2. 기초 자산에 대한 **명확한 법적 권리** 체계 확립
  3. **실제 소유권의 온체인 이전** (RWA 모델)

**Backed Finance TSLAx와의 차이**
- Synthetix: 가격만 추종, 법적 권리 없음
- Backed: 실제 테슬라 주식 1:1 담보, 법적 권리 보장

### 1.2 RWA의 이점과 잠재력

#### 1.2.1 국경·화폐·시간에 제약받지 않는 결제

**전통 금융 vs 온체인 결제 비교**

| 구분 | 전통 금융 | 온체인 (스테이블코인) |
|------|----------|---------------------|
| **송금 시간** | 3~5 영업일 | 수 분 |
| **수수료** | 평균 6.6% | 3% 이하 (최대 95% 절감) |
| **운영 시간** | 영업시간 제한 (주말·공휴일 불가) | 24/7 (연중무휴) |
| **환율 리스크** | 높음 (보유 기간 길어짐) | 실시간 정산으로 최소화 |
| **중개기관** | 다수 은행 거침 | 불필요 |

---

**아토믹 결제 (Atomic Settlement)**

**개념**
```
자산 이전 + 대금 지급을 하나의 트랜잭션으로 묶음
→ 둘 중 하나라도 실패하면 전체 자동 원상복구
```

**효과**
- 거래상대방 위험(Counterparty Risk) 원천 제거
- "All or Nothing" 원칙
- 불완전 결제 방지

---

**실제 사례: ANZ 은행 (Chainlink CCIP 활용)**

**배경**
- 호주·뉴질랜드 은행(ANZ)
- 크로스체인 금융 거래 실험

**프로세스**
```
1. A 블록체인에서 NZD → AUD 변환
2. B 블록체인의 AUD로 토큰화 탄소배출권(Carbon Credit) 구매
3. Chainlink CCIP Messaging으로 A 블록체인에서 탄소배출권 보유 기록
4. 스마트 컨트랙트가 아토믹 결제 보장
```

**의의**
- 다중 블록체인 환경에서도 안전한 결제
- 실시간 환전 + 자산 구매 동시 처리
- 전통 금융기관의 온체인 금융 참여 사례

---

#### 1.2.2 통합된 인프라: 유동성 증대, 절차 간소화, 가격 발견 효율성 증대

##### 전통 금융의 비효율: 파편화 (Fragmentation)

**문제점**
- 원장, 결제 시스템, 청산 기관, 예탁 기관 모두 분리
- 수많은 중개기관의 장부 대조 필요
- 시간 소요, 비용 증가
- 오류 가능성

---

##### RWA의 해결책: 블록체인 단일 인프라

**1. 유동성 증대**

**자산 분할(Fractionalization)**
- 부동산, 미술품 등 고가 자산을 소액 단위로 분할
- 소액 투자자 참여 가능
- 투자자 기반 확대

**글로벌 24/7 시장**
- 언제 어디서든 거래 가능
- 시간대 제약 없음
- 지역 제한 없음

**2. 절차 간소화**

**계약 내재화**
- 스마트 컨트랙트에 모든 규칙 코드화
- 배당금 지급 자동 실행
- 이자 지급 자동 실행
- 의결권 행사 자동 기록

**아토믹 결제**
- 자산 이전 + 대금 지급 동시 완결
- 중개 단계 최소화

**3. 가격 발견 효율성 증대**

**정보 비대칭성 해소**
- 모든 거래가 투명한 원장에 기록
- 실시간 정보 접근
- 시장 조작 어려움

**WEF (세계경제포럼) 분석**
> "스마트 컨트랙트 기반 토큰화 → 유동성·가격 발견·거래 신속성 개선"

#### 4.1.1 조각투자 시장의 급성장

4. **피스**: 부동산 조각투자

---

#### 4.1.2 조각투자의 법적 성격

**현행 법적 분류**
- 신탁 수익증권
- 자본시장법상 증권
- BUT 블록체인 기반 아님 (중앙 데이터베이스)

**한계**
- 진정한 의미의 RWA가 아님
- 2차 시장 유동성 제한적
- 플랫폼 종속성 높음

---

#### 4.1.3 시사점

**수요 입증**
- 한국에도 RWA 투자 수요 존재
- 부동산, 지적재산권 등 다양한 자산 선호

**제도화 필요성**
- 투자자 보호 장치 미흡
- 2차 시장 활성화 필요
- 블록체인 기반 진정한 토큰화 요구

---

### 4.2 법제화 이후: RWA 시장 전망과 주요 과제

#### 4.2.1 한국 토큰증권 시장 개요

**법제화 현황**
- 자본시장법 개정안 (2023.2 발표)
- 전자증권법 개정안
- 금융위원회 토큰증권 가이드라인

**토큰증권 정의**
- 분산원장에 기록·관리되는 증권
- 발행, 유통이 블록체인에서 이루어짐

---

#### 4.2.2 발행·유통 분리 원칙의 한계

**규제 원칙**
- 토큰증권 발행 플랫폼 ≠ 유통 플랫폼
- 이해상충 방지 목적
- 발행사는 자사 발행 증권 중개 금지

**시스템 관점의 문제점**

##### [1] 기술적 복잡성 증가

**다기관 분산원장 동기화 문제**
| 문제 영역 | 세부 내용 |
|----------|----------|
| **정합성** | 발행 원장 ↔ 유통 원장 간 데이터 일치 보장 필요 |
| **결제 실패 처리** | 거래 실패 시 롤백 및 책임 소재 불명확 |
| **트랜잭션 확정** | 체결 완료 데이터 실시간 수신 구조 부족 |
| **시퀀스 관리** | 거래 시점과 체결 순서 일관성 유지 어려움 |
| **결함내성** | 일부 참여자 장애 시 전체 영향 리스크 |
| **네트워크 보안** | 복수 기관 연결 시 통신보안 위협 |

**운영 리스크**
- 단일 플랫폼 대비 **훨씬 높은** 트랜잭션 동기화 요구
- 메시지 표준화 필요
- 장애 복구 시나리오 복잡

---

##### [2] 투자자 UX 저하

**중복 온보딩 문제**
```
1. 발행 플랫폼 A: 회원가입 + KYC + 투자적합성 확인
2. 유통 플랫폼 B: 회원가입 + KYC + 투자적합성 확인 (다시)
→ 이탈률 증가
```

**상태 추적 불투명성**
- 청약 → 배정 → 입고 → 매매 → 권리행사 → 상환
- 라이프사이클이 2개 플랫폼에 분산
- 투자 여정 전체 데이터 통합 어려움
- 퍼널 분석, 이탈률 진단 불안정

**사례 분석의 어려움**
- 발행 측과 유통 측의 데이터 기준 상이
- 이벤트 순서 꼬임
- 누락·중복 문제
- 인과추론 신뢰도 저하

---

##### [3] DvP (Delivery versus Payment) 구현 난도 증가

**문제**
- 증권 계좌 (발행 플랫폼) ≠ 현금 결제 계좌 (유통 플랫폼)
- 두 플랫폼 간 정산 분절

**결과**
1. 체결 대비 현금 반영 지연 (미수·미납)
2. 부분결제/부분인도 정합성 문제
3. 배당·쿠폰 등 현금흐름 경로 분산
4. 잔고 확인 복잡화
5. 증권 대사처리 부담 증가

---

#### 4.2.3 토큰증권 인프라 구축

##### 4.2.3.1 한국형 토큰증권을 위한 금융 인프라 필수 요건

**클라우드 BaaS (Blockchain as a Service) 아키텍처 핵심 요건**

| 요건 | 세부 내용 |
|------|----------|
| **규모 확장성** | 사용자 증가에 따라 선형 확장 |
| **낮은 지연시간** | 거래 응답과 상태 반영 보장 |
| **DR (재해복구)** | 장애 상황에서도 서비스 연속성 |
| **모니터링** | 운영·감독 관점의 실시간 관찰 |
| **보안 체계** | 금융사고 방지 |
| **표준 인터페이스** | 다기관 연계 |

---

**금융위 가이드라인 및 실무 요건**

**합의 알고리즘**
- BFT (Byzantine Fault Tolerance) 방식
- 빠른 Finality 보장

**스마트 컨트랙트**
- ERC-1400 표준
- 권리·문서·보호예수 등 증권 업무 포괄

**네트워크 업그레이드**
- 중단 최소화
- 호환성 유지 원칙

**Private Key 관리**
- 네트워크 참여자 역할 기반 권한 분리
- 노드 아키텍처: 다기관 분산 + 접근통제

---

##### 4.2.3.2 인프라 아키텍처

**1. 사용자 수에 따른 규모 확장성**

**수평 확장 (Scale-Out) 전략**

**로드 밸런서 (Load Balancer)**
- 클라이언트 요청을 여러 서버로 분산
- 헬스 체크로 비정상 인스턴스 자동 우회
- 다중 AZ (Availability Zone) 지원

**고가용성 확보**
- 특정 AZ 장애 시 자동 트래픽 전환
- 데이터 저장 계층 다중 AZ 이중화
- SPOF (Single Point of Failure) 제거

**오토스케일링 (Auto Scaling)**
- CPU, 메모리, 네트워크 메트릭 모니터링
- 임계치 초과 시 인스턴스 자동 증설·축소
- 트래픽 변동성 능동 대응

---

**2. 데이터베이스 규모 확장 (Replication)**

**Primary/Replica 구조**
- **Primary**: 쓰기 (Insert/Update/Delete) 전담
- **Replica**: 읽기 (Select) 담당
- 조회 부하 수평 분산

**다중 AZ 배치**
- 장애 시 데이터 일관성 유지
- 서비스 연속성 보장

---

**3. 캐시 (Cache)**

**Redis 인메모리 캐시**
- 값비싼 연산 결과 메모리 저장
- 후속 요청 지연 감소

**블록체인 트랜잭션 처리**
- Write Tx: 상태 변경
- Read Tx: 상태 조회

**비동기 파이프라인**
```
API Gateway → SDK/서비스 → Kafka 메시지 큐 → 캐시 → RPC 노드
```

**백그라운드 데몬**
- 주기적으로 캐시 ↔ 원장 대조
- TXID 확정 감지
- Kafka/WebSocket으로 체결 완료 이벤트 통지

---

**4. DR (Disaster Recovery)**

**고가용성 목표: 99.99%**

**다중 AZ (Availability Zone)**
- 물리·논리적으로 분리된 데이터센터
- 특정 AZ 헬스 체크 실패 시 자동 우회
- Region 간 백업·복제로 광역 재해 대비

**자동 장애조치 (Failover)**
- 상태 저장 컴포넌트 (DB, 체결 엔진) 자동 승격
- 주 인스턴스 → 보조 인스턴스 권한 전환
- 서비스 연속성 보장

---

**5. Monitoring**

**Prometheus + Grafana**
- 블록 확정 지연
- 큐 적체
- 실패·재시도율
- 노드 상태
- RPC 가용성

**Alert 목적별 분리**
- 운영용
- 보안용
- 감독용

---

##### 4.2.3.3 발행 및 유통 플랫폼 구조

**발행 구조**

**프로세스**
1. 투자자 → 계좌관리기관 증권 계좌 개설/연동
2. 표준 API로 분산원장 주소 생성, 계좌번호와 1:1 매핑
3. 계좌관리기관(증권사) 또는 발행인 → 컨소시엄 원장 노드 참여
4. 발행사 → 계좌관리기관에 상품 등록 요청
5. 투자자 → 상품 청약
6. 청약 종료 → 발행사가 상품 등록(토큰 배포) 요청
7. 투자자 → 배정 물량 확정
8. 분산원장에 투자자 권리 등록
9. 발행사 → 콘솔/API로 원장 상태 확인, 보유 수량 조회

---

**유통 구조**

**DvP 구현을 위한 메시지 큐 + Deposit Token(DT) + Security Token(ST) + Lock 메커니즘**

**프로세스**
1. **결제계좌 개설/연동**
   - 유통 플랫폼에서 증권사 계좌 개설 또는 연동

2. **주문 접수 & 현금 토큰화 (DT)**
   - 매수/매도 주문 제출
   - 원화 → DT로 교환 (Mint/Hold)
   - 오더북에 주문 등록
   - 결제 전용 토큰으로 격리

3. **보유 확인 (ST 잔고 조회)**
   - 장외거래 중개업자가 컨소시엄 원장 조회
   - 매도자 ST 보유 수량 검증

4. **증거금 잠금 (Lock/Escrow)**
   - 스마트 컨트랙트 Lock 기능
   - DT와 ST 증거금 잠금
   - 잠금 미설정 시 체결 대상 제외

5. **체결 후 DvP 실행 (MQ 기반 오케스트레이션)**
   - 호가 일치 → 체결
   - 장외거래 중개업자 → MQ로 결제·권리이전 지시
   - 계좌관리기관 → 자체 권한으로 'swap()' 호출
   - ST ↔ DT 원자적 교환
   - 멀티시그 등 정책 적용

**예외·롤백 규칙**
- **원자성**: DT 또는 ST 어느 한쪽이라도 커밋 실패 시 잠금 자동 해제, 롤백
- **멱등성**: 멱등성 키로 중복·역순 도착 흡수, 재시도/타임아웃 관리
- **대사 (Reconciliation)**: 체결·권리·정산 이벤트 일괄 대사, 잔고·총량 정합성 보장

---

#### 4.2.4 결론

**한국 토큰증권의 잠재력**
- 투자계약증권·수익증권 중심 초기 단계
- 제도 성숙 시 채무·지분·파생결합·예탁증권까지 확장 가능
- 일본 사례: 제도 정비 + 실질 수요 → 기하급수적 성장

**발행·유통 분리 원칙의 도전**
- 구현 난도 높음
- 기관 레거시 ↔ 온체인 동기화
- 중복 온보딩 UX 마찰
- DvP 안정적 구현 병목

**성공 방정식**
```
규제 정합성 + 다기관 연동성 + 운영 안정성 = 성공적 인프라
→ 발행-유통-결제 프로세스 일관 연결
→ 콘텐츠 IP, 채권, 펀드 등 실질 수요 자산군 해외 투자자 제공
→ 원화 스테이블코인 매개로 디지털 자산 시장 확대
```

**과제**
1. 다기관 연동 인프라 표준 업계 합의·정립
2. 합리적 규제 설계
3. 정교한 인프라 구현

## PART 5. 실물자산 토큰화의 법적 구조와 국내 규제 제안

### 저자
김효봉 (법무법인(유) 태평양 파트너 변호사)

---

### 5.1 실물자산 토큰화의 법적 구조

#### 5.1.1 서론

**토큰화의 잠재력과 법적 과제**
- 국경 초월 거래
- 프로그래머블 특성
- 금융 혁신 가능성

**법적 불확실성**
- 토큰 자체: 전 세계 자유롭게 거래
- 기초자산 권리: 해당 자산 관할권 법 제도 구속
- 실질적 권리 이전 수반 토큰화 → 법 제도 면밀 고려 필수

---

#### 5.1.2 토큰화 구조의 유형

**3가지 유형**
1. **직접 토큰화 (Direct Tokenisation)**
2. **간접 토큰화 (Indirect Tokenisation)**
3. **불완전 토큰화 (Incomplete Tokenisation)**

---

##### 5.1.2.1 직접 토큰화

**정의**
- 토큰 자체가 자산의 법적 권리 구현
- 토큰 = 원시 증권/권리증서와 동일 효력
- 토큰 보유자 = 기초자산 법적 권리자

**적용 자산**

**1. 금융상품**
- 펀드 지분, 채권 등 유가증권을 토큰 형태 발행
- 자본시장법 전면 적용
- 토큰 이전, 공·사모 발행, 권리 행사 규제 준수

**2. 동산 재산권**
- 토큰 = 동산 소유권 증서
- 한계: 물리적 자산 상태 변경(훼손, 분실) 정보 갱신 어려움

**3. 부동산 권리 증서**
- '디지털 등기권' 개념
- 토큰 = 부동산 소유권 증서 동일 효력
- 실시간 부동산 거래 가능
- 시도 관할권: 에스토니아, 슬로베니아, 조지아

**4. 계약상 권리**
- 소유권, 수익참가권, 이용권 등
- 토큰 보유 = 계약 당사자 지위
- 스마트 컨트랙트/별도 계약서에 명시

**5. 신규 금융상품 창출**
- 매출채권 토큰화 (Tokenised Receivables)
- 서비스 이용 혜택 토큰
- 블록체인 상 이전 = 권리 양도 효력/대항 요건

---

##### 5.1.2.2 간접 토큰화

**정의**
- 중간 법적 권리층(layer) 신설
- 기초자산 보유/참조하는 신규 금융상품 발행
- 그 신규 자산 형태로 토큰 발행
- **자산담보토큰 (Asset-Backed Token)**

**구조**
- 기초자산과 토큰화 자산 별개 공존
- 투자자: 경제적 가치만 간접 취득
- 기초자산 직접 소유권 취득 어려움

**활용 경우**
1. 법 제도 미비로 직접 토큰화 불가 → 우회 수단
2. 중개구조로 맞춤형 투자상품 제작 (전략적)

---

**주요 중개 구조**

**1. 특수목적회사 (SPV)**
- 부동산 소유 회사 설립
- 회사 지분을 토큰으로 발행
- 예: 미국 RealT (Delaware LLC 활용)

**2. 규제된 펀드**
- 부동산·주식·채권 등 편입 투자펀드
- 펀드 지분 토큰화
- 자본시장법 또는 EU UCITS 지침 적용
- 수탁인(custodian) 자산 보관 및 리스크 관리 의무

**3. 신탁업자 (Trustee) 구조**
- 신탁업자가 기초자산 보유
- 권리 표창 토큰을 온체인 발행
- 투자자에게 교부

**4. 등기소 (Registrar) 이중 원장**
- 전통 원장 + 블록체인 원장 병행 운영
- 양측 기록 동기화

**평가 기준**
1. 기초자산 보관·관리 방식
2. 평가의 투명성
3. 토큰 보유자 청구권 보호 강도

---

##### 5.1.2.3 불완전 토큰화

**정의**
- 토큰이 기초자산 권리를 온전히 구현하지 못함
- 부수적 수단 (Auxiliary)에 머무름
- 법적 구속력 있는 권리 미담보

**기능**
- 정보 공유
- 자산 관리
- 사실 증명
- '디지털 쌍둥이 (Digital Twin)'

---

**유형 1: 계약상 증빙 토큰**

**사례: HSBC Gold Token**
- 토큰 보유 ≠ 실물 금 직접 소유권
- 토큰 자체 = 아무 권리나 가치 미대표
- 금융상품(파생상품)에 대한 소유 증표
- 투자자: 금값 연동 계약상 청구권만 보유

**국내 적용 시 리스크**
- 자본시장법 직접 적용 회피 → 불완전 토큰화 검토 가능
- 규제 리스크 존재 (감독당국 용인 여부 불명확)

**평가 기준**
1. 토큰 시스템 기술적 정확성·신뢰성
   - DLT 무결성, 거버넌스, 투명성, 감사가능성
2. 법적 청구권 보호
   - 계약 상대방 이행 신뢰
   - 토큰 정보, 서면 약정, 이행 절차, 관할권 법의 강제력

---

**유형 2: 미러링 토큰**

**정의**
- 법률상 1차 권리 기록: 기존 장부/시스템
- 토큰: 보조적 표시 수단

**문제점**
1. 법적 권리: 원래 시스템에만 유효
2. 토큰 보유자 권리 주장 → 기존 기록 따라야 함
3. 이중 기록 시스템 문제
   - 일관성 유지 (동시 갱신)
   - 권한 관리
   - 정보 기밀성

**분쟁 리스크**
- 토큰의 법적 한계 명확 고지 미흡
- 사용자가 토큰 = 실제 재산권 오인
- 법적 분쟁 발생 가능성

**국내 사례**
- 전자증권법 개정 전: 블록체인 미러링 거래 임시 허용
- 제도 정비까지의 과도기적 조치

#### 5.2.2 유럽연합 (EU): 2023년 DLT Pilot Regime

**법적 근거**
- Regulation (EU) 2022/858
- 유럽 디지털금융 전략의 일환

**목적**
1. DLT 활용 금융상품 (Tokenized Securities) 발행·거래·결제 실험
2. DLT 기반 새로운 시장 인프라 개발
3. 투자자 보호·시장 무결성·금융 안정·투명성 유지
4. 규제차익 없이 혁신 유도

---

**DLT 시장인프라 유형 (3가지)**

**1. DLT MTF (다자간매매체계)**
- 투자회사 또는 시장운영자 인가 기관 운영
- DLT 금융상품만 취급
- 기존 주식시장과 유사한 거래 플랫폼

**2. DLT SS (결제시스템)**
- CSD 인가 기관 운영
- DLT 기반 증권결제시스템
- 토큰화 증권 결제 및 기록 (Book-Entry)

**3. DLT TSS (거래·결제 통합시스템)**
- MTF + CSD 기능 통합
- 새로운 유형 인프라
- 거래 체결 + 동시결제 (DvP) 구현

---

**허가 및 운영 기간**

**특별허가**
- 파일럿 기간에만 유효
- 2023년 3월 ~ 최소 3년, 최대 6년 연장 가능

**종료 기한**
- 별도 입법조치 없으면 자동 일몰 없음
- 후속 입법 전까지 현행 체계 지속

---

**허용 DLT 금융상품**

| 자산 유형 | 규모 한도 |
|----------|----------|
| **주식** | 시가총액 5억 유로 미만 |
| **채권·채무증권** | 발행 규모 10억 유로 미만 |
| **UCITS 지분** | AUM 5억 유로 미만 |

**인프라 규모 한도**
- 단일 DLT 인프라 총 취급: **60억 유로 (약 10조 원)**
- 초과 시 신규 자산 추가 불가
- **90억 유로 (약 15조 원)** 도달 시: 전환 전략 발동, 감독 통지

---

**신청 자격**

**기존 금융기관**
- DLT MTF: MiFID II 투자회사 또는 시장운영자 인가 보유
- DLT SS: CSDR CSD 인가 보유
- DLT TSS: 투자회사, 시장운영자, CSD 모두 신청 가능

**신규 진입자**
- 투자회사 또는 CSD 인가를 조건부 취득
- DLT 파일럿 허가 동시 신청
- 임시 인가 (파일럿 기간에만 유효)
- 파일럿 종료 시 정식 인가 전환 또는 사업 정리

---

**규제 면제 (핵심 조항)**

**1. MiFID II 참여자 제한 완화**
- 현행: MTF 회원 = 인가 금융기관만
- 파일럿: 비금융투자자 (개인 등) 직접 참여 가능 (승인 조건 하)
- 조건: 평판, 거래경험, DLT 지식, 고빈도매매·시장조성 금지

**2. MiFIR 거래보고 의무 면제**
- 현행: 모든 거래 규제당국 보고 (MiFIR 제26조)
- 파일럿: 승인 시 면제
- 이유: DLT 장부에 거래기록 자동 보관
- 조건: 거래내역 전부 기록 보관, 규제당국 실시간 열람 권한 제공
- 방식: 규제 옵저버 노드로 참여

**3. CSDR 계좌 및 결제 의무 예외**
- 현행: 모든 증권 중앙예탁기관 명의 계좌 기록·결제 (계좌식 등록)
- 파일럿: 분산원장 보유기록으로 대체 가능
- 조건:
  - 모든 토큰화 증권 DLT 기록
  - 총발행량 = 실제 발행량 일치
  - 참가자별 보유분 구분 기록
  - 무결성 확보 (임의 초과발행·삭제 금지)

**4. 결제자금 방식 예외**
- 현행: 중앙은행 예금 원칙
- 파일럿: 은행 예금 토큰, 전자화폐 토큰 (e-money token) 허용
- 조건: 결제수단 리스크 식별 및 관리

---

**추가 요건 (강화된 기술 관리)**

**Regulation (EU) 2022/858 Article 7**

| 요건 | 세부 내용 |
|------|----------|
| **사업계획** | 명확한 규칙 문서화 |

| **자산·키 관리** | 고객 자산 분리보관, 콜드스토리지, 다중서명, 접근통제 |

**사례: 21X (독일 DLT TSS, 2024)**

#### 5.2.3 리히텐슈타인: 2019년 TVTG (토큰 및 신뢰기술 서비스제공자 법)

**입법 배경**
- 2018년 법제 작업 시작
- 2019년 10월 3일 의회 만장일치 통과
- 2020년 1월 1일 발효

**목적**
- 토큰과 권리의 디지털 표현·이전 민사법 기반 마련
- 고객·자산 보호
- 토큰 경제 참여 서비스 제공자 권리·의무·감독 규정

---

**토큰 정의 (TVTG 제2조)**

**정의**
- 신뢰기술(TT) 시스템 상의 정보
- 청구권, 회원권, 재산권 등 절대적·상대적 권리 대표 가능
- 하나 이상의 TT 식별자에 할당

**특징**
- 토큰 = 디지털 정보 단위
- 현실 세계 법적 권리·자산 표상
- 토큰이 반드시 기초자산 보유 필요 없음 (권리 없는 토큰도 존재 가능)

**역할**
- 권리의 컨테이너 (Container)

---

**토큰의 법적 지위**

**리히텐슈타인 법 적용 시**
- 토큰 = 리히텐슈타인 소재 자산
- 재산권적 지위 부여
- 토큰이 담는 권리와 분리되면서 독립적 법적 취급

**증권 여부**
- 금융상품에 해당 시 별도 금융시장법령 병행 적용
- TVTG 자체: 포괄적 디지털 권리증서로 규정

---

**토큰 컨테이너 모델**

**정의**
- 토큰 = 권리의 직접 구현물 아님
- 토큰 = 권리를 담는 그릇

**담을 수 있는 권리**
- 계약상 청구권
- 회원권
- 지식재산권
- 물권적 권리

**기능**
- 디지털 환경에서 권리 이전·거래 매개

---

**소유권 및 처분권**

**처분권 (Right of Disposal)**
- 토큰 처분권자 = 토큰 대표 권리의 적법 귀속자

**TT 키 보유자**
- 사실상 토큰 지배
- 처분 능력자 (Power of Disposal)
- 원칙: 키 보유자 = 법적 처분권자 (소유자) 추정

**이전 보유자**
- 보유 당시 정당한 처분권자로 추정

**효과**
- 디지털 토큰 실제 보유 상태 ↔ 법적 권리 귀속 일치
- 블록체인 거래 기록 = 권리 공시 기능

---

**토큰 양도 (처분) 요건**

**처분 정의**
- 토큰 처분권 (소유권) 이전
- 제한적 물권 (용익권, 담보권) 설정

**유효 조건**

**1. 기술적 이전**
- TT 시스템 규칙에 따라 유효하게 이동
- 블록체인 거래가 프로토콜상 유효하게 확정
- 제한물권 설정 시: 이전 없이 시스템상 표시로 효력 발생

**2. 당사자 합의**
- 양도인·양수인 간 처분권 이전/제한물권 설정 합의
- 명시적 선언
- 계약법적 합의

**3. 정당한 권리자에 의한 처분**
- 양도인 = 적법한 처분권자 (소유자)
- 무권리자 처분: 무효 또는 취소 가능

---

**토큰 이전과 기초권리 이전 연결**

**TVTG 제7조**
> "토큰에 대한 처분은 토큰이 대표하는 권리에 대한 처분의 효과를 갖는다"

**효과**
- 토큰 이전 = 기초권리 이전
- 법적 연결고리 마련

##### (2) 금융시장법 분야 개정

**중앙예탁기관 없이 증권 발행·유통 가능**

**선택 가능**
- 희망 시 토큰증권을 기존 예탁결제 시스템 예탁 가능
- 기존 계좌증권과 동일 지위 부여

---

**Security Token (토큰화 증권) 발행 법제화**

**정의**
- 블록체인에 기록된 디지털 증권

**발행 가능 증권**

- 주식

- 안전한 기술적 요건 갖춘 블록체인 등록

#### 5.3.5 금 토큰화

**핵심 개념**
- 실물자산 담보 스테이블코인
- 1 토큰 = 1온스 순금 가치
- 발행사: 동일 양의 물리적 금 보관

**특징**
- 24시간 글로벌 거래
- 미세단위 분할소유
- 물리적 금으로 상환 가능

---

**사례 1: PAX Gold (PAXG)**

| 항목 | 세부 내용 |
|------|----------|
| 발행사 | Paxos (NYDFS 인가 신탁회사) |
| 출시 | 2019년 |
| 담보 | 1 PAXG = LBMA 인증 1온스 금괴 |
| 소유권 | 특정 골드바 소유권 |
| 블록체인 | Ethereum (ERC-20) |
| 분할 | 0.000001온스까지 가능 |
| 최소 구매 | 제한 없음 |
| 감사 | 월간 회계감사 |
| 인출 | 금괴 또는 현금 교환 |
| DeFi | Aave, Curve 담보 활용 |
| 보관수수료 | 연 0.02% (ETF 대비 절감) |

**사례 2: Tether Gold (XAU₮)**

| 항목 | 세부 내용 |
|------|----------|
| 발행사 | Tether |
| 출시 | 2020년 |
| 담보 | 1 XAU₮ = 1온스 금 (스위스 금고) |
| 블록체인 | ERC-20, TRC-20 |
| 특징 | 토큰당 특정 금괴 할당, 번호 조회 |
| 규제 | 미국 직접 규제 받지 않음 |
| 투명성 | PAXG 대비 낮음 (정기 감사 미흡) |

---

**법적 성격**

**보관증서 (Warehouse Receipt) 또는 상품파생 증권**

**Paxos 구조**
- 신탁회사로서 1:1 금 보유
- 토큰 소유자 = 금 소유권 인정

**증권 여부**
- 논쟁 존재
- 기초자산 금 가격 연동 외 별도 수익 없음
- 실물 금 청구권 기능 → 상품성 평가
- SEC 현재까지 별도 제재 없음

**기술**
- 월간 감사 공개
- 토큰-금괴 매칭 조회 툴
- 스마트 컨트랙트: 소유자 주소 변경, 수수료 징수

### 5.4 국내 법제화 제안

#### 5.4.1 토큰화로 인한 신규 리스크

**미국 의회조사국 (CRS) 보고서 (2024.5.20)**

**규제 체계 권고**
> "토큰화 = 새로운 자산 창출 아님, 현행법 대응 가능. BUT 온체인 거래 특성 반영 제도 보완 필요"

**미국 현황**
- 토큰 증권 전용 법제 없음
- 기존 증권법 활용
- 대부분 Reg D → 전문투자자만 접근
- 법적 불확실성 존재 (글로벌 공통 문제)

---

**주요 리스크**

**1. 시스템 리스크**
- 결제·거래 효율 향상
- BUT 금융 스트레스 시 유동성 위험 증폭
- 순식간에 뱅크런 촉발 가능

**2. 운영 리스크**
- 퍼블릭 블록체인: 24/7 운영, 탈중앙화
- 해킹·오류 발생 시 책임 소재 불명확

**3. 법률 구조 및 권리 문제**
- 스마트 컨트랙트 코드 ↔ 법적 계약 불일치 위험
- 코드 오류 → 배당 잘못 집행 시 법적 책임 문제

**4. 기술적 리스크**
- 스마트 컨트랙트 버그
- 해킹
- 키 분실
- 코드 오류 → 자산 동결/오류 거래
- 해킹 → 토큰 탈취 → 실물자산 청구권 행사 문제

**5. 프라이버시 문제**
- 퍼블릭 체인: 모든 거래 내역 공개
- 해결책: 영지식증명 (ZKP), 프라이빗 블록체인
- BUT 구현 복잡, 브릿지 해킹, 상호운용성 문제

#### 5.4.2 국내 규제 제안

##### (1) 규제 샌드박스 시행

**필요성**

- 안전장치 제도화 방안 불명확

**제안**
- 본격 법제화 전 다양한 토큰화 프로젝트 시범 운영
- 리스크 탐지 → 제도 설계 반영

**수단**
- 현행 규제 샌드박스 활용

**전제 조건**
- 관련 법령 개정 (아래 참조)

---

##### (2) STO 법안 개정 필요

**현행 문제점**

**미국 BlackRock BUIDL**
- Securitize 플랫폼 발행
- Reg D 사모 면제
- 지갑으로 유통
- 거래소/지갑 앱 매수, DEX 이용 가능

**vs 국내 STO**
- 극도로 제한적

**금융위 2023.2.6 보도자료 분석**

| 항목 | 제한 내용 | 문제점 |
|------|----------|--------|
| **토큰화 대상** | 비금전 신탁 수익증권, 투자계약증권만 | 주식·채권·펀드 (시장 수요 많음) 제외 |
| **공모 간주** | 토큰화 증권 = 공모 발행 간주 | 사모 면제 적용 불가 |
| **블록체인** | 대통령령 정하는 분산원장만 | 프라이빗 블록체인만 가능 (확장성 제한) |
| **소액공모 한도** | 30억 원 또는 100억 원 | 미국 대비 턱없이 낮음 |

**개정 필요 사항**

**시행령 위임 내용 재검토**
1. **토큰화 대상 확대**: 정형증권 포함
2. **사모 면제 허용**: 토큰화 증권도 사모 면제·소액공모 활용 가능
3. **공모 간주 재고**: 토큰화 = 공모 간주 규정 철회
4. **소액공모 한도 상향**: 다양한 발행 실험 허용
5. **퍼블릭 블록체인 허용**: 글로벌 표준 호환성

---

##### (3) 금융규제 혁신법안 개정 필요

**토큰 자산 자연스러운 유통**
- 지갑 통한 이전
- 스테이블코인 결제

**문제**
- 발행인이 지갑 서비스 (스왑·커스터디) 제공
- 스테이블코인 취급
- → 가상자산사업자 (VASP) 해당 가능성
- → 특금법 제7조 신고 의무

---

**규제 샌드박스 활용 시 장애물**

**현행 금융혁신지원 특별법 제2조 '금융관련법령'**
- 별표 금융관계 법률
- 대통령령 정하는 법령

**문제**
- 가상자산이용자보호법 미포함
- 특정금융정보법 미포함

**해결책**
- 동 법 및 시행령 개정
- 샌드박스 대상 법령에 추가
  - 가상자산이용자보호법
  - 특정금융정보법

---

##### (4) 사법상 블록체인 법적 효력 근거 규정 마련

**장기 과제**
- 민사상 블록체인 법적 효력 법리 정리
- 민사법 반영

---

**주요 쟁점**

**1. '지배(Control)' 개념 도입**
- UNIDROIT 디지털자산 및 사법 원칙
- '지배' 개념 국내 민사법 수용 여부
- 선의취득, 담보권 설정 법률효과 인정

**2. 토큰 ↔ 실물자산 연동 (Link)**
- 토큰 ≠ 자동 실물자산 물권 이전 보장
- 온체인 기록 ↔ 오프체인 법률관계 조화 과제

**3. 공시 및 소유권 충돌**
- 토큰 ≠ 기존 등기·인도 대체 불가
- 블록체인 ↔ 기존 공시제도 연계
- 실물자산 ↔ 디지털 토큰 소유권 불일치 해소

**4. 기초자산 상태·가치 투명성**
- 상태, 담보, 권리관계, 시가 변동
- 오라클 (Oracle) 또는 제3자 검증 기관 활용

**5. 선의취득 및 담보권 설정**
- 디지털자산 자체: 선의취득·담보권 가능
- BUT 실물자산 연계 효과: 국내법 별도 입법 필요

---

**종합 결론**

**RWA 토큰화 법적 효력 인정 필요 조치**
1. 블록체인 기록 = 법적 유효 공시 방식/권리 변동 수단 인정
2. 법제 정비 필요
   - 민법 공시 원칙
   - 등기제도
   - 물권변동 규정

**재산법 체계 재검토**
- 전통 재산법 전면 재검토

**단기 vs 장기**
- 단기: 법인 통한 간접 토큰화
- 장기: 전통 등기 ↔ 블록체인 연동, 스마트 컨트랙트 법적 효력 인정 등 제도 혁신

**지속 연구**
- 민사법 체계 전반 심도 연구
- 입법 논의 지속

## 📊 종합 결론

- 법제화 지연
- 규제 프레임워크 불명확
- 글로벌 경쟁에서 뒤처짐 위험

**성공을 위한 핵심 과제**
1. **토큰증권(STO) 법제화 조속 추진**
   - 토큰화 대상 확대 (정형증권 포함)
   - 사모 면제 허용
   - 소액공모 한도 상향
   - 퍼블릭 블록체인 허용

2. **원화 스테이블코인 발행 전략**
   - 통화 주권 확보
   - RWA 세틀먼트 기축통화
   - 디지털 달러라이제이션 방어

3. **규제 샌드박스 활성화**
   - 금융혁신법 개정 (가상자산법, 특금법 포함)
   - 다양한 RWA 프로젝트 실험
   - 리스크 탐지 및 제도 반영

4. **온체인 금융 인프라 구축**
   - BaaS 아키텍처
   - 다기관 연동 표준
   - DvP 안정적 구현
   - 발행-유통-결제 일관 연결

5. **민사법 정비**
   - 블록체인 법적 효력 근거
   - 토큰 ↔ 실물자산 연동 법리
   - 공시제도 현대화

**비전**
- 디지털 금융 시대의 '설계자 (Architect)'
- '디지털 G2' 달성
- 새로운 금융 프로토콜 설계 주도

**선택의 기로**
- **현재 지연**: 디지털 금융의 '사용자 (User)'로 전락
- **적극 추진**: 글로벌 리더로 도약
